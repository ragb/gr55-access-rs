# Bootstrap prompt: `gr55-access-rs` (Roland GR-55 guitar synth tooling)

This document is a starter prompt + plan for opening a fresh Claude Code session in an empty directory and building tooling for the **Roland GR-55 guitar synthesizer**. It distills the patterns and methodology that worked on this `re202-access-rs` repo.

The GR-55 shares Roland's SysEx conventions with the RE-202: same DT1/RQ1 command pair (`0x12` / `0x11`), same checksum formula `(128 - (sum mod 128)) mod 128`, same 4-byte addresses sent MSB-first, same `F0 41 [device-id] [model-id] [cmd] [addr*4] [data...] [chk] F7` framing. **But the model ID is different** — the GR-55 uses the 3-byte sequence `00 00 53` between device-id and command, where the RE-202 uses the 5-byte `00 00 00 00 18`. It is also much **larger** — it has full instrument patches with PCM tones, COSM modeling, multi-effects, system settings, and many more parameters per slot — so the protocol shape will be familiar but the address map and data model will be considerably bigger.

These framing facts (model ID, command bytes, header shape) were cross-checked against [sixeight7/VController's `MIDI_GR55.ino`](https://github.com/sixeight7/VController/blob/master/MIDI_GR55.ino), an open-source MIDI controller project with a working GR-55 driver. Treat that source as a useful sanity check throughout — it's hardware-tested code, not a paper spec.

### Dependency choices

- **`midir`** for MIDI I/O. The `MidiSession` pattern: own one `MidiOutputConnection`, one `MidiInputConnection<()>` whose callback forwards bytes to an `mpsc::channel`, drain pending before each request, filter inbound to only matching DT1 frames at the requested address. Handle MIDI Thru echo by address-matching, not by timing.
- **`clap`** with `derive` for the CLI. Global args (`--port`, `--input-port`, `--output-port`, `--device-id`) on `Cli`; subcommands as a single enum.
- **`serde`** + **`serde_yaml`** with `#[serde(rename_all = "snake_case")]`. Use `#[serde(untagged)]` enums for fields that can be either a symbolic value (e.g. `"off"`) or a numeric one — gives a human-friendly YAML surface.
- **`schemars`** for JSON Schema emission, behind a `schema` feature flag.
- **`tsify-next`** for TS types, behind a `tsify` feature flag.
- **`wasm-bindgen`** + **`wasm-pack`** for the JS surface. `crate-type = ["cdylib", "rlib"]`.
- **`anyhow`** in the CLI for error wrapping, **`thiserror`** in `core` and `midi.rs` for typed errors.
- **`log`** + **`env_logger`** in the CLI.

### Roland SysEx framing (shared with GR-55)

| Field | Value | Notes |
|---|---|---|
| SOX | `F0` | |
| Manufacturer | `41` | Roland |
| Device ID | `10`..`1F` (UI 1..16); `7F` = broadcast | confirm GR-55's default on hardware |
| Model ID | **`00 00 53`** (3 bytes) | shorter than RE-202's 5-byte `00 00 00 00 18`. Verified via [VController source](https://github.com/sixeight7/VController/blob/master/MIDI_GR55.ino). |
| DT1 (write) | `12` | confirmed identical to RE-202 |
| RQ1 (read) | `11` | confirmed identical to RE-202 |
| Address | 4 bytes, each 0..=`7F`, MSB first | |
| RQ1 payload | 4-byte size, 7-bit nibbles | same encoding as RE-202 |
| Checksum | `(128 - ((sum(addr+data)) mod 128)) mod 128` | identical formula to RE-202 |
| EOX | `F7` | |

The full GR-55 frame on the wire is `F0 41 [device-id] 00 00 53 [cmd] [addr*4] [data...] [chk] F7`. The `Frame` type in `gr55-core/src/sysex.rs` is a near-straight port of RE-202's — just change the model ID constant from `[0x00, 0x00, 0x00, 0x00, 0x18]` to `[0x00, 0x00, 0x53]` and adjust the length-derived offsets accordingly.

**Family code in the Universal Identity Reply** is still unverified — VController doesn't query identity, the public MIDI Implementation Chart doesn't list it, and no community write-up I found includes the reply bytes. Capture it from the device early.

### CLI surface to mirror

```
gr55 ports                            # list MIDI ports
gr55 identity                         # Universal Identity Request, print parsed reply
gr55 dump  --patch N    -o file.yaml
gr55 dump  --system     -o file.yaml
gr55 dump  --edit       -o file.yaml  # if GR-55 has an edit-buffer mirror
gr55 dump  --all        -o dir/       # bundles system + patches into a directory
gr55 sync  --patch N    -i file.yaml [--verify]
gr55 sync  --all        -i dir/       [--verify]
gr55 diff  a.yaml b.yaml              # file-vs-file textual diff
gr55 show  file.yaml                  # pretty-print + identify which model
gr55 lint  file.yaml                  # validate through the typed model
gr55 select N                         # PC# on Rx channel — confirm PC#-to-patch mapping
gr55 schema system|patch              # emit JSON Schema
```

`--verify` (read-back-after-write + byte compare) caught silent write failures on RE-202; bring it over from day one.

### CI

Two jobs in `.github/workflows/ci.yml`: native (Ubuntu, with `apt-get install -y libasound2-dev` for midir's ALSA backend) and wasm. The native job runs fmt, clippy with `-D warnings`, tests, and a **schema-drift check** that regenerates `schemas/*.json` and `diff -u`s against the committed copies. The wasm job runs `wasm-pack test --node` and `wasm-pack build --target web --release`, uploads `pkg/`.

`env: FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: "true"` at the top of the workflow opts into Node 24 ahead of the June 2026 forced switch.

## What's different on GR-55 (research these first)

The RE-202's data model is small (one 33-byte memory block per slot, plus 18 bytes of system). GR-55 is dramatically larger — it has layered patches with multiple instrument parts (PCM tones, COSM modeling, normal pickup, etc.), full multi-effects, separate memory banks. Concrete things that differ:

1. **Model ID `00 00 53`** (3 bytes) — verified above against VController. Send Universal Identity Request (`F0 7E 7F 06 01 F7`) early anyway to capture the **family code** and **software-revision bytes** (the reply has `41 [family-lo family-hi] [number-lo number-hi] [4-byte sw-rev]`); these are still unverified from public sources and you want them recorded per-fixture.
2. **Patch count**: GR-55 ships with **297 user patch slots**; firmware v1.50 expanded the **factory** patches to 468 (360 guitar, 108 bass). The Rust `PatchSlot` analogue should accommodate that range. Source: [Sound Programming overview](https://soundprogramming.net/guitar-synthesizers/roland/roland-gr-55/) and the v1.50 release notes referenced on vguitarforums.
3. **A larger address tree** with sub-areas inside patches (e.g. `<patch-base> + <subsystem-offset> + <param-offset>`). Roland documents nest these by indentation in the PDF; the address map will need a multi-level type. The full byte size of one patch is **not documented in public sources** — measure it from a real dump.
4. **More than one bank/category of patches** (USER and PRESET; the firmware divides PRESET into guitar/bass categories). The `MemorySlot` analogue may need to be richer than RE-202's `Manual | User(u8)`.
5. **Possibly a temporary-patch / edit-buffer at a different address** than RE-202's `20 00 00 00`. Sweep to find it the same way: enumerate prefixes, RQ1 each, see what replies with bytes that change when you change patch on the device. VController writes to a "current patch" area but doesn't isolate the address — the sweep will tell you.
6. **Tap tempo / BPM encoding is not documented in public sources** — may or may not use the same 4-nibble MSB→LSB packing as RE-202. Verify by writing a known value and reading it back.
7. **Firmware-version sensitivity**: GR-55 v1.0 vs v1.50 introduced new factory patches but no documented MIDI map breakage. Record the sw-rev bytes from Identity Reply in every fixture filename so future-you can tell which firmware a fixture came from.
8. **CC map** will be much bigger and more category-segmented than RE-202.

Don't enumerate the GR-55 address map by hand from the PDF before capturing fixtures — visual tables in Roland PDFs use indent-as-hierarchy and dot/checkmark-as-membership, and an LLM summary of those tables **will get cell membership wrong**. Either:

- Capture the raw HTML/PDF and grep markers (`&Aacute;`, `&bullet;`, etc.) yourself, OR
- Cross-reference your model against a real device dump.

This bit you on RE-202 when reconstructing Mode head combinations from an LLM summary — saved as the `parse-visual-tables-raw` feedback memory.

## Methodology that worked

These are listed in execution order.

### 1. Get the spec, but distrust it until verified

Find the **official MIDI Implementation PDF**. For RE-202 the canonical Roland host didn't have it; a third-party mirror (`zikinf.com`) did. Don't waste time hunting on roland.com if a community mirror has the same PDF.

Then validate the spec against the hardware before building anything substantial on it. Even authoritative-looking specs contain mistakes or omit useful behavior (RE-202's edit-buffer mirror is not documented at all).

### 2. Capture fixtures before writing the data model

The most useful thing you can do on day one is dump 5–10 representative slots/areas off the real device and save them as `.syx` byte files in `gr55-core/tests/fixtures/`. Examples from RE-202:

- `system_dump.syx` — full system area
- `memory_001_dump.syx` — default-ish slot
- `memory_002_dump.syx` — user-programmed slot with real values
- `memory_manual_dump.syx` — MANUAL slot
- `edit_buffer_mirrors_memory2.syx` — edit-buffer snapshot proving it mirrors active memory

These let you write round-trip tests **before** you have any opinions about field layout. If `from_bytes(&fixture).unwrap().to_bytes().unwrap() == fixture`, the codec is at least internally consistent.

### 3. Build typed-model-with-named-enums YAML format

Don't go straight from raw bytes to YAML. Define typed Rust structs/enums with `serde(rename_all = "snake_case")` and untagged enums for symbolic-or-numeric fields. The YAML surface is then the serde projection of the typed model, which means:

- The schema is autogenerated and stays in sync (schemars feature).
- Editor validation works (`# yaml-language-server: $schema=...` header).
- TS types are autogenerated for the wasm surface (tsify-next feature).

For ranged parameters that have a value + min/max for expression-pedal sweeps, use a nested struct:

```yaml
repeat_rate:
  value: 63
  min: 63
  max: 63
```

Not flat `repeat_rate: 63 / repeat_rate_min: 63 / repeat_rate_max: 63`. Nested reads better and the type derivation handles it for free.

### 4. Discover the edit buffer by address sweep

Roland devices commonly expose a mirror of the currently-active patch at some address — writes are volatile (live-edit), reads always reflect the active slot. Find it by:

1. List all already-known addresses (system base, patch bases).
2. Enumerate plausible unused prefixes (`00`, `01`, `02`, ... `1F`, `20`, `30`, `40`, ...).
3. For each prefix, RQ1 with a moderate size (say 16 or 32 bytes) and record any address that replies with data identical to a known patch slot.
4. Confirm by changing the active patch on the device between probes — the mirror's content should change.

`tools/explore/probe.py sweep` in this repo does this. Port it to the GR-55 session.

### 5. CLI before WASM

Build a working `gr55 dump` / `gr55 sync` against the real device first. The wasm crate is a thin re-export of `gr55-core` and should not block CLI iteration.

### 6. Use Python (`mido`) for exploration, Rust for the shipped tool

Throughout RE-202 development, `tools/explore/` was where ad-hoc hardware probing happened — and it stayed in Python on purpose. The relevant axes:

- **Iteration speed.** Editing a small `mido` script and re-running is faster than `cargo build` + binary rerun, especially during the address-sweep / "is this byte alive?" phase when you're trying ten things in five minutes.
- **No data-model dependency.** Python scripts work with raw bytes from day one. The Rust `Frame` codec, typed model, and CLI take a few iterations to stabilize; Python doesn't need any of them to be working in order to dump a slot and grep the bytes.
- **Chicken-and-egg fixture capture.** You need real `.syx` dumps to write codec tests against, but you can't dump cleanly until the codec exists. Python breaks the cycle: `capture.py` writes raw `.syx` files that the Rust tests later assert against.

What lives in `tools/explore/`:

- `.venv/` (gitignored), `requirements.txt` — just `mido` and a MIDI backend.
- `probe.py` — single RQ1 (`rq1`), single DT1 (`dt1`), and an address sweep (`sweep`). Reproduce this verbatim for GR-55 with the new model ID constant.
- `sweep_addresses.py` — exhaustive prefix walker; used to find the edit buffer.
- `capture.py` — dumps known regions to `.syx` files for fixture creation.
- `settle_open_questions.py` — one-off scripts each tied to a specific protocol question. Disposable; commit them anyway as a record of what was tested.
- `probe_current_memory.py` / `probe_current_memory_sweep.py` — recent additions: drive the device through several memories via PC# and read a candidate "current memory" register at each step to see whether the byte tracks.

Conventions that worked:

- Single helper `roland_checksum(bytes) -> int` defined identically in every script (don't bother factoring — these are throwaway probes).
- Same `build_rq1(device_id, address, size)` / `build_dt1(device_id, address, data)` shape; copy-paste between scripts is fine.
- `pick(names, needle)` for substring-matching port names (don't hardcode port names; the user's setup varies).
- Read replies by filtering for the matching DT1 (`0x12`) at the requested address — ignore echoes from MIDI Thru, ignore MIDI clock spam from active tap-tempo.
- Print one line per request/reply in hex; keep output greppable.

When something *is* worth keeping permanently, port the logic to Rust and add a test against a fixture. Python is the lab notebook; Rust is the published paper.

### 7. Confirm against hardware aggressively

Every protocol claim should ultimately be tested. From RE-202:

- Identity reply: confirmed actual bytes.
- Edit-buffer mirror: confirmed by switching active memory between RQ1s.
- Writability of edit buffer: confirmed by DT1 + audible change.
- Slot persistence: confirmed by power-cycle + read-back.
- Time Mode precedence: confirmed by writing one, reading the other, comparing.
- PC# mapping: confirmed by user observation (PC#0 = MANUAL, not "unknown").

Each of these resolved a question that would otherwise have lingered as ambiguity in the docs.

## Risks / things to watch on GR-55 specifically

- **Address space is much larger.** RE-202 has ~5 distinct base prefixes. GR-55 likely has dozens. The `address.rs` types may need to be generated or table-driven rather than hand-coded.
- **Patch data is layered.** A "patch" on GR-55 is composed of subsystems (PCM tone 1, PCM tone 2, COSM, normal pickup, multi-effects with chain ordering, ...). The Rust model will probably want a `Patch { common: PatchCommon, tone_1: PcmTone, tone_2: PcmTone, cosm: Cosm, effects: Effects, ... }` shape. Each subsystem deserves its own module.
- **Effect chain ordering** is a discriminated-union-of-effect-blocks problem. Plan to use `#[serde(tag = "type")]` enums for effect slots.
- **Larger dumps mean slower full-device round-trips.** `dump --all` on RE-202 reads 130 small blocks; on GR-55 each patch may be kilobytes. Consider batching or progress indication.
- **Firmware versions matter.** GR-55 has shipped multiple firmware versions over the years; the address map may differ between them. Capture the firmware version (Identity Reply software-revision bytes) into every fixture filename and the dump frontmatter.

## Status note (added during this session)

The original bootstrap above assumed direct GR-55 hardware access; the actual
project starts with **no device** (a friend has one but is not a developer,
and is only available for a one-shot GUI session). The fixtures-before-code
gate has been replaced by **mining FloorBoard's `midi.xml` address map as
primary truth**, cross-checked against the Roland owner's manual PDF and
[VController](https://github.com/sixeight7/VController/blob/master/MIDI_GR55.ino).
Wire-byte verification uses FloorBoard's bundled `default.syx`, `system.syx`,
and `EZ-Tone.syx` files rather than live captures. See `docs/sysex-notes.md`
for the current per-claim provenance and `MEMORY.md` (gitignored, in
`~/.claude`) for the constraints driving this shift.
